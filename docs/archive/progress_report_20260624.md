## 1. 概要（プロジェクトの目的）

### 1.1 このプロジェクトで作っているもの

**ドローンを遠隔で操作・監視するための地上管制システム（Ground Control Station = GCS）**です。

簡単に言うと、**パソコンやブラウザからドローンの状態を見たり、飛行指示を出したりできるシステム**です。

### 1.2 なぜ必要なのか

| 課題 | 解決策 |
|---|---|
| 複数のドローンを同時に管理したい | 1つの画面で全機の状態を一覧表示 |
| GPS精度をcm級にしたい | RTK補正技術で高精度測位を実現 |

### 1.3 システム全体像

```
┌─────────────┐     Tailscale(VPN)     ┌──────────────┐     UART     ┌──────────┐
│  PC/Mac     │ ◄──────────────────► │  Raspberry Pi │ ◄─────────► │ Pixhawk  │
│  (GCS画面)  │    インターネット経由    │  (通信中継)    │   シリアル   │ (ドローン)│
└─────────────┘                       └──────────────┘             └──────────┘

補足:
- Tailscale = 安全なVPN。離れた場所の機器を同じネットワーク内のように接続できる技術
- Raspberry Pi = 小型コンピュータ。ドローンの近くに設置し、通信を中継する役割
- Pixhawk = ドローンの頭脳。GPSやセンサーの情報を集め、飛行を制御する
- UART = シリアル通信規格。Raspberry PiとPixhawkを有線で接続
```

### 1.4 主な機能

| 機能 | 説明 |
|---|---|
| 📡 **テレメトリ受信** | ドローンのGPS位置・高度・速度・バッテリー残量などをリアルタイム表示 |
| 🎮 **機体制御** | アーム（プロペラ始動）・離陸・着陸・移動指示を遠隔から実行 |
| 🛰️ **RTK高精度測位** | GPSの誤差をcm級に補正する技術。専用アンテナ（u-blox）を使用 |
| 📊 **マルチドローン対応** | 複数機を1つの画面で同時に管理 |
| 🌐 **ブラウザ対応** | Webブラウザだけで操作可能。ソフトのインストール不要 |

### 4.2 RTK補正システム

RTK（Real-Time Kinematic）は、GPSの誤差を**cm級**まで改善する技術です。

**仕組み:**
1. 位置が正確にわかっている基準局（u-bloxアンテナ）が「GPS衛星の信号は今これだけずれている」という補正データを生成
2. その補正データをドローンのGPSに送信
3. ドローン側が補正を適用 → 誤差が1m→1cmに改善

**構築済みの流れ:**
```
u-blox → シリアル通信 → RTK基地局 → TCP:2101 → 屋内サーバー → MAVLink → ドローン
```

#### u-blox F9P の動作モード：Survey-In と TIME mode の違い

サンプル1〜3では水平誤差を算出できたが、サンプル4〜5では算出できなかった理由は、u-blox ZED-F9P の**動作モードの違い**によるものです。

| 項目 | Survey-In mode | TIME mode |
|------|------|------|
| **目的** | 基準局の**正確な絶対位置を計測**する | RTCM補正データを**生成・出力**する |
| **動作内容** | GPS信号を長時間（例：600秒）受信し、位置を平均化して高精度に確定 | 確定済みの基準位置を使って、各衛星の誤差量（補正データ）を計算 |
| **NMEA出力** | ✅ **あり**（$GPGGA等、現在の推定位置を出力） | ❌ **なし**（位置は固定済みのため出力不要） |
| **出力内容** | NMEAセンテンス + RTCM（設定による） | RTCMバイナリのみ |
| **位置情報** | 徐々に収束中（精度向上中） | 確定済み（不変） |

**なぜ NMEA 出力の有無が重要なのか：**

- 水平誤差 = PixhawkのGPS座標 − u-blox基準局の座標
- この計算には **u-bloxの現在位置（NMEAデータ）** が必須
- TIME mode では NMEA が出力されない → **基準局の座標が取得できない** → 誤差計算不可 🔴

```
Survey-In mode:  u-blox → NMEA出力あり → 基準位置がわかる → 誤差計算可能 ✅
TIME mode:       u-blox → NMEA出力なし → 基準位置がわからない → 誤差計算不可 🔴
```

**サンプルごとのモード状態：**

| サンプル | u-bloxモード | NMEA出力 | 誤差算出 |
|------|------|------|------|
| 1 | 通常DGPS動作 | ✅ あり | ✅ 1.13m |
| 2 | 通常DGPS動作 | ✅ あり | ✅ 20.35m ⚠️ |
| 3 | Survey-In直後（TIME mode移行途中） | ✅ 一時的にあり | ✅ 2.62m |
| 4 | TIME mode（完全移行後） | ❌ 停止 | 🔴 算出不可 |
| 5 | TIME mode（完全移行後） | ❌ 停止 | 🔴 算出不可 |

> **サンプル3のみ例外**：Survey-In 完了直後で TIME mode に移行しつつも、NMEA 出力が完全には停止しておらず、偶然 u-blox の位置データが取得できたと推測されます。
> 
> **サンプル4〜5**では TIME mode への移行が完全に完了し、RTCMバイナリ出力専用となったため、u-blox の位置データが一切取得できなくなりました。

**推奨対策：** u-blox を再度 Survey-In mode に切り替えることで NMEA 出力が再有効化され、基準局座標との絶対誤差比較が可能になります（次回アクション 🔴1, セクション7.6 参照）。
