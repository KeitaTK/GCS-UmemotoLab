## 🔴 優先度1: RTK fix (fix=5/6) 未達の原因分析レポート作成

### 背景

前回（6/30）のサンプル4取得時、fix=4 (DGPS) が 9,450回観測されたが、fix=5 (RTK_FLOAT) および fix=6 (RTK_FIXED) は **0回** であった。
RTCM補正データの注入自体は成功している（fix: 3→4 遷移で証明）にも関わらず、RTK ambiguity（整数値バイアス）が解決できていない。

### 1-1. 原因分析ドキュメントの作成

`docs/rtk_fix_failure_analysis.md` を作成し、以下の構成で原因を体系的にまとめる。

#### A. 症状の整理
- [x] fix=4 (DGPS) は安定して観測（3ログ合計 1,540回: gcs.log.3=61, .4=728, .5=751）
- [x] fix=5 (RTK_FLOAT) は過去全ログで 0回
- [x] fix=6 (RTK_FIXED) は過去全ログで 0回
- [x] RTCM注入は正常（type 269/271/281/307 注入確認済み）
- [x] 衛星数 31〜32（十分）

#### B. 推定原因（階層構造で記述）

```
RTK fix=6 未達の根本原因
│
├── 【第1層】F9P が DroneCAN 接続（GPS_TYPE=9）
│   ├── UART接続（GPS_TYPE=17）ではない
│   ├── RTCMデータはPixhawk経由で GPS_RTCM_DATA (msg#67) として注入される
│   └── → F9P側のRoverモード設定がCAN経由で正しく届いているか不明
│
├── 【第2層】RoverモードのRTCM入力設定が不完全
│   ├── F9P RoverはRTCM3入力を受け付けるモードに明示的に設定される必要がある
│   ├── GPS_AUTO_CONFIG=2 ではDroneCAN自動設定を試行するが成功は未確認
│   └── → QGC DroneCANメニューでF9PノードのTMODE設定を直接確認する必要あり
│
├── 【第3層】整数値アンビギュイティ（Integer Ambiguity）未解決
│   ├── RTK Fixedに至るには、搬送波位相の整数値バイアスを解決する必要がある
│   ├── 必要な条件: ①共通衛星5個以上 ②基準局- Rover間距離が短い ③連続したRTCMデータ
│   ├── 本環境: ①衛星31-32個（OK）②同一屋内（距離~0m, OK）③RTCM注入は断続的（⚠️）
│   └── → RTCMデータの連続性不足がambiguity解決を妨げている可能性
│
└── 【第4層】RTCM TCP接続の断続性
    ├── localhost化（6/30）でTailscale切断問題は解消
    ├── しかし基地局のシリアル競合・プロセス蓄積問題が残存
    └── → 長時間安定したRTCMストリームの維持が課題
```
