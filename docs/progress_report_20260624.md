## 1. 概要（プロジェクトの目的）

### 1.1 このプロジェクトで作っているもの

**ドローンを遠隔で操作・監視するための地上管制システム（Ground Control Station = GCS）**です。

簡単に言うと、**パソコンやブラウザからドローンの状態を見たり、飛行指示を出したりできるシステム**です。

### 1.2 なぜ必要なのか

| 課題 | 解決策 |
|---|---|
| 複数のドローンを同時に管理したい | 1つの画面で全機の状態を一覧表示 |
| GPS精度をcm級にしたい | RTK補正技術で高精度測位を実現 |

### 1.3 システム全体像

```
┌─────────────┐     Tailscale(VPN)     ┌──────────────┐     UART     ┌──────────┐
│  PC/Mac     │ ◄──────────────────► │  Raspberry Pi │ ◄─────────► │ Pixhawk  │
│  (GCS画面)  │    インターネット経由    │  (通信中継)    │   シリアル   │ (ドローン)│
└─────────────┘                       └──────────────┘             └──────────┘

補足:
- Tailscale = 安全なVPN。離れた場所の機器を同じネットワーク内のように接続できる技術
- Raspberry Pi = 小型コンピュータ。ドローンの近くに設置し、通信を中継する役割
- Pixhawk = ドローンの頭脳。GPSやセンサーの情報を集め、飛行を制御する
- UART = シリアル通信規格。Raspberry PiとPixhawkを有線で接続
```

### 1.4 主な機能

| 機能 | 説明 |
|---|---|
| 📡 **テレメトリ受信** | ドローンのGPS位置・高度・速度・バッテリー残量などをリアルタイム表示 |
| 🎮 **機体制御** | アーム（プロペラ始動）・離陸・着陸・移動指示を遠隔から実行 |
| 🛰️ **RTK高精度測位** | GPSの誤差をcm級に補正する技術。専用アンテナ（u-blox）を使用 |
| 📊 **マルチドローン対応** | 複数機を1つの画面で同時に管理 |
| 🌐 **ブラウザ対応** | Webブラウザだけで操作可能。ソフトのインストール不要 |

### 3.2 通信構成

```
【屋外】                         【MacBook Air（GCS+RTK基地局）】
                                  
┌──────────┐     SSHトンネル       ┌──────────────┐
│ Raspi +  │ ◄────Tailscale───── │ GCSサーバー   │
│ Pixhawk  │                      │ :8000 (Web UI)│
│ (UART接続)│                      │              │
└──────────┘                      │              │
                                  ├──────────────┤
u-blox ──USB── localhost ────     │ RTK基地局     │
                                  │ :2101 (RTCM)  │
                                  └──────────────┘
```

**特徴:**
- Raspi ↔ MacBook Air: **Tailscale** VPN + SSHトンネルで安全に通信（大学ネットワーク越え可能）
- u-blox ↔ GCS: **USB直結 + localhost**でTailscale切断問題を完全解消
- GCSはlaunchdにより常時稼働、再起動後も自動復活

### 4.2 RTK補正システム

RTK（Real-Time Kinematic）は、GPSの誤差を**cm級**まで改善する技術です。

**仕組み:**
1. 位置が正確にわかっている基準局（u-bloxアンテナ）が「GPS衛星の信号は今これだけずれている」という補正データを生成
2. その補正データをドローンのGPSに送信
3. ドローン側が補正を適用 → 誤差が1m→1cmに改善

**構築済みの流れ:**
```
u-blox → シリアル通信 → RTK基地局 → TCP:2101 → 屋内サーバー → MAVLink → ドローン
```

### 4.3 RTCM注入の接続安定性 調査結果

| 調査項目 | 結果 |
|---|---|
| RTCM注入成功 | ✅ 17:02:56に確認（type 281/307/1018/268/269/271, 669 bytes/s） |
| 接続維持時間 | ❌ 数分で切断（`Connection refused`） |
| 再接続 | ❌ 指数バックオフ再試行するも全て失敗 |
| データ収集中のRTCM | ❌ サンプル3取得時は不通 |

**原因**: macbook-air上のRTCM基地局(TCP:2101) 間の TCP 接続が Tailscale DERP リレー経由（遅延 600-1100ms）で確立するが、数分後に切断され再接続できない。

**対策**: 以降のサンプルではRTCM接続を事前確認し、接続から1分以内にデータ取得を完了する。

### 5.4 RTCM接続安定性の問題 🔴 重要課題

#### 現象

**GCSサーバー → RTCM基地局間間のTailscale TCP接続が数分で切断され、再接続不可となる問題**がサンプル2〜4で継続的に発生している。

| サンプル | 現象 | 影響 |
|---|---|---|
| サンプル2 | RTCM注入確認後、数分で切断 | GPS絶対位置異常（系統誤差20m） |
| サンプル3 | データ取得中（約16サンプル目）に切断 | 実質DGPS状態で測定完了 |
| サンプル4 | TBD（対策後検証待ち） | — |

#### 通信経路

```
macbook-air (u-blox)                  MacBook Air (GCS+RTK)
┌──────────────────┐     Tailscale    ┌──────────────┐
│ rtk_base_station │ ──── TCP:2101 ──→│ RtcmReader    │
│   .py            │                 │   → Injector  │
│                  │                 │   → Pixhawk   │
└──────────────────┘                 └──────────────┘
```

#### 推定原因

| # | 仮説 | 可能性 |
|---|---|---|
| 1 | Tailscale TCP keepalive 不足 | 🔴 高 |
| 2 | NAT/ファイアウォールのタイムアウト | 🔴 高 |
| 3 | RTCM Reader再接続ロジックの不備 | 🟡 中 |
| 4 | 大学ネットワークの中間機器によるTCPセッション切断 | 🟡 中 |
